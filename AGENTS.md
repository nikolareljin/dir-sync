# Repository Guidelines

## Project Structure & Module Organization
- `src/dirsync/` contains the application code (tray app, sync logic, scheduling, notifications).
- `tests/` holds pytest suites like `test_*.py`.
- `scripts/` provides local helpers for linting, testing, and packaging.
- `docs/` contains build and testing references.
- `dist/` is generated by build steps and should not be edited manually.

## Build, Test, and Development Commands
- `python -m venv .venv && source .venv/bin/activate`: create and activate a local virtualenv.
- `pip install -e .[dev]`: install the project and dev tooling.
- `python -m dirsync.app` or `make run`: run the tray app locally.
- `make lint` or `scripts/lint.sh`: run Ruff and Black checks.
- `make test` or `scripts/run_tests.sh`: run the pytest suite.
- `make build` or `scripts/build.sh`: build a standalone binary; output goes to `dist/`.
- `./scripts/test_in_docker.sh` and `./scripts/build_in_docker.sh`: run tests/builds in Docker.

## Coding Style & Naming Conventions
- Python 3.10+, formatted with Black (line length 100) and linted with Ruff (`E`, `F`, `I`, `B`).
- Use `snake_case` for modules and functions, `CapWords` for classes, and `UPPER_SNAKE_CASE` for constants.
- Keep new files under `src/dirsync/` and tests under `tests/` with `test_*.py` naming.

## Testing Guidelines
- Framework: `pytest` with optional `pytest-cov`.
- Default discovery lives in `tests/`; keep unit tests small and focused.
- For UI flows, follow the smoke checklist in `docs/TESTING.md` after changes that affect the tray app.

## Commit & Pull Request Guidelines
- Commit messages follow a lightweight conventional pattern: `feat:`, `chore:`, `wip:` + short present-tense summary.
- PRs should include: summary of changes, test commands run (or reason skipped), and screenshots when UI behavior changes.
- Link related issues or feature requests when applicable.

## Configuration & Packaging Notes
- Configuration import/export uses YAML; keep example files sanitized of real paths or secrets.
- Packaging steps are documented in `docs/BUILD.md`; prefer scripts to keep builds consistent.
